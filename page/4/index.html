Java异常总结

发表于 2018-03-07 | 更新于 2019-09-15 | 分类于 Java

一、总结

二、概述

1、Java异常类图

  1. Error:JVM 系统内部错误、资源耗尽等严重情况

  2. Exception:

    1. RuntimeException（运行时异常）是指因设计或实现方式不当导致的问题。产生这种异常的特点是Java编译器不会检查它，也就是说程序，程序中可能出现这类异常时，即使没有用try-catch语句捕获它，也没有用throws语句声明抛出它，还是会编译通过的。由于没有处理它，当出现这类异常时，异常对象一直被传递到main()方法，则程序将异常终止。如果采用了异常处理，异常将会被相应的程序执行处理。
    2. CheckedException(检查异常)是RuntimeException以外的异常，类型上都属于Exception类及其子类。从程序语法角度讲是必须进行处理的异常，如果不处理，程序就不能编译通过。

  3. 用户自定义异常：如果要自定义异常类，则扩展Exception类即可，因此这样的自定义异常都属
    于检查异常（CheckedException）。如果要自定义非检查异常，则扩展自RuntimeException。
    按照国际惯例，自定义的异常应该总是包含如下的构造函数：

    • 一个无参构造函数
    • 一个带有String参数的构造函数，并传递给父类的构造函数。
    • 一个带有String参数和Throwable参数，并都传递给父类构造函数
    • 一个带有Throwable参数的构造函数，并传递给父类的构造函数。

2、五个关键字

  1. throws:用来标明一个成员函数可能抛出的各种”异常”。
  2. throw:用来明确地抛出一个”异常”。
  3. try:包含可能导致异常的保护代码。
  4. catch:捕获保护代码的异常。
  5. finally:确保一段代码不管发生什么”异常”都被执行一段代码。

throws和throw的区别
throws：用来声明一个方法可能产生的所有异常，不做任何处理而是将异常往上传，谁调用我我就抛给谁。

  • 用在方法声明后面，跟的是异常类名
  • 可以跟多个异常类名，用逗号隔开
  • 表示抛出异常，由该方法的调用者来处理
  • throws表示出现异常的一种可能性，并不一定会发生这些异常

throw：则是用来抛出一个具体的异常类型。

  • 用在方法体内，跟的是异常对象名
  • 只能抛出一个异常对象名
  • 表示抛出异常，由方法体内的语句处理
  • throw则是抛出了异常，执行throw则一定抛出了某种异常

3、先逮小的，再逮大的

即当捕获异常时，父类异常的捕获语句不可以写在子类异常的捕获语句的上面。

4、异常和重写的关系

  1. 父类没有声明异常，那么子类也不能声明异常；
  2. 重写的方法不能比被重写的方法抛出更宽泛的异常；
  3. 抛出异常类型的数目不可以比原有的方法抛出还多；

抽象类和接口

发表于 2018-03-07 | 更新于 2019-09-11 | 分类于 Java

一、抽象类（abstract修饰符）

  1. abstract修饰的类为抽象类，此类不能有对象(无法对此类进行实例化，即不能new)；
  2. abstract修饰的方法为抽象方法，此方法不能有方法体(就是什么内容都不能有)；
  3. 有抽象方法的类一定是抽象类，但抽象类中不一定有抽象方法；
  4. 抽象类中的抽象方法必须在子类中被实现，子类实现方法必须含有相同或更低的访问级别，
    抽象类的子类具体实现父类的所有抽象方法，否则也是抽象类；

二、接口(interface)

  1. 接口中的所有属性默认为:public static final;
  2. 接口中的所有方法默认为：public abstract;
  3. 接口不能被实例化；
  4. 接口只能包含方法声明；
  5. 接口的成员包括方法、属性、索引器、事件；
  6. 接口中不能包含常量、字段(域)、构造函数、析构函数、静态成员；
  7. 接口不再像类一样用关键字extend去”继承”，而是用implements去”实现”;
  8. 一个类可以实现多个接口，但只能继承于一个类，所以可以通过这种方式来实现多继承；

三、接口和抽象类的区别

  1. 抽象类描述的是”是不是”的问题，而接口描述的是”有没有”的问题；
  2. 接口强调特定功能的实现，而抽象类强调所属关系；
  3. 抽象类中可以有构造方法，接口中不能有构造方法；
  4. 接口可以被多重实现，抽象类只能被单一继承；

四、接口和抽象类的相同点

  1. 都可以被继承
  2. 都不能被实例化
  3. 都可以包含方法声明
  4. 派生类必须实现未实现方法

面向对象的设计思想

发表于 2018-03-06 | 更新于 2019-09-10 | 分类于 Java

一、面向对象语言三大特征

  1. 封装
  2. 继承
  3. 多态

二、设计思想

  1. 看到一个需求时不应该直接写代码，应该先考虑有哪些类；
  2. 考虑类的时候，类一定是一类事物的描述，不能太局限；
  3. 考虑类的时候需要考虑主要的类，也就是需要和业务、动作、事件紧密相关的类；
  4. 属性：不需要把该类的所有属性都声明出来，只需要自己用到的即可；
  5. 属性一般会私有，提供get、set方法区操作；
  6. 方法：考虑封装方法的时候，就站在该类的立场去考虑；
  7. 隐藏和封装：减低耦合度；
  8. 要增加或强化功能的时候，首先应该想的是扩展（重写或重载等），而不是修改；
  9. 当同等类型的、相似类型的类过多的时候，应该想到继承；
  10. 继承多了的时候可以采用多态的方式，用父类引用指向子类对象，从而解决了处理多
    个对象而产生的不同方法了
  11. 因为继承是很耦合性的，且继承只能继承一个类，所以如果要继承多个父类的情况就
    要用接口实现。

多态总结以及动态绑定机制

发表于 2018-03-06 | 更新于 2019-09-10 | 分类于 Java

一、多态

1、概念

父类引用指向子类对象（内存堆区的地址）

2、多态实现的必要条件

  • 存在继承关系
  • 存在方法重写
  • 父类引用指向子类对象

3、多态的体现

  • 继承体现了多态
  • 接口体现了多态

4、生活中的多态

关于多态的例子不胜枚举，比方说按下F1键这个动作，如果当前在Flash界面下弹出的就是

AS3的帮助文档；如果当前在Word下弹出的就是Word帮助；在Windows下弹出的就是Windows

帮助和支持。同一个事件发生在不同的对象上会产生不同的结果。

5、多态访问成员

  1. 访问成员变量

编译时期看左边的类型,如果左边类型中没有该变量则编译报错

程序运行时期看左边类型,左边类型的变量的值就是运行的结果

  1. 访问成员方法

编译时期看左边的类型,如果左边类型中没有该方法则编译报错

程序运行时期看右边类型,即程序会执行右边类型的方法

  1. 访问构造方法

多态访问子类构造方法时会先访问父类构造方法

我们可以利用这一特性来帮助子类初始化父类继承过来的成员

  1. 访问静态方法

编译时期看左边的类型,如果左边类型中没有该方法则编译报错

程序运行时期看右边类型,即程序会执行右边类型的方法

6、多态的优点

  • 简化代码
  • 提高程序的扩展性和可维护性

7、多态的缺点

使用父类引用无法直接访问子类所特有的成员，需要向下转型，即父类引用转为子类对象（强制转

换），且向下转型的过程中如果没有转换为真实子类类型，会出现类型转换异常（ClassCastExcep

tion），故每次在向下转型之前需要做一个类型判断。

二、动态绑定机制

1、概念

动态绑定是指在执行期间判断所引用对象的实际类型，根据其实际类型调用其相应的方法

2、程序绑定的概念

绑定指的是一个方法的调用与方法所在的类（方法主体）关联起来，对Java来说绑定分为静态绑定

和动态绑定，或者叫做前期绑定（编译时绑定）和后期绑定（运行时绑定）

3、区分

  1. 静态绑定

程序在JVM运行过程中，会把类的类型信息、static属性和方法、final常量等元数据加载到方法区

这些在类加载时就已经知道，不需要对象的创建就能访问的，就是静态绑定的内容；

  1. 动态绑定

需要对象创建出来，使用时根据队中的实例对象的类型才进行取用的就是动态绑定的内容。

Java中类的实例化顺序

发表于 2018-03-05 | 更新于 2019-09-13 | 分类于 Java

总结

  1. 先静态、先父后子。 先静态：父静态 > 子静态 。
  2. 优先级：父类 > 子类 ， 静态代码块 > 非静态代码块 > 构造函数。

例题分析(数字为执行顺序)

例一：

    package Servlet.day0811.am;
    public class StaticTest {
    /*1、静态代码块先加载，当new一个新对象时调用构造方法且该类的非静态代码块要执行一次
    注意：(静态代码块与非静态代码块的区别，静态代码块只在第一次new的时候执行一次,
        而非静态代码块在每次new的时候都要执行一次)*/
    static StaticTest st=new StaticTest();  
    //5、按顺序执行静态代码块 
    static{
        System.out.println("静态代码块");
    }
    //3、构造方法执行
    public StaticTest(){
        System.out.println("构造器");
        System.out.println(st);
        m1();
    } 
    //4、m1方法被调用  7、m1方法再次被调用
    public static void m1(){
        System.out.println("m1静态方法");
        //执行到顺序7时，构造方法已经完全执行完，即st对象被实例化
        System.out.println(st);
    }
    {
        //2、此非静态代码块执行(因为非静态构造方法的优先级高于构造方法)
        System.out.println("普通代码块");
    }
    //6、按顺序执行静态代码块
    public static void main(String[] args) {
        System.out.println("main开始");
        StaticTest.m1();
        //8、按顺序执行
        System.out.println("main结束");
    }
}

输出结果

普通代码块 （先加载普通代码块）
构造器 （再加载构造器）
null （输出构造器中的st，此时并未加载静态属性，所以输出null）
m1静态方法 （构造器中的m1方法，输出m1静态方法）
null （m1方法中的st，此时st仍然未被加载）
静态代码块 （加载静态代码块）
main开始 （main方法开始，st在这时候被加载（分配内存））
m1静态方法 （调用静态方法m1）
Servlet.day0811.am.StaticTest@d1f24bb （st已经被加载过了，所以为对象对应的地址）
main结束 （main方法结束）

Java中ArrayList和LinkedList的区别

发表于 2018-03-05 | 更新于 2019-09-17 | 分类于 Java

一、大致区别

  1. ArrayList是实现了基于动态数组的数据结构，LinkedList基于链表的数据结构。

    （LinkedList是双向链表，有next也有previous）

  2. 对于随机访问get和set，ArrayList觉得优于LinkedList，因为LinkedList要移动指针。

  3. 对于新增和删除操作add和remove，LinedList比较占优势，因为ArrayList要移动数据。

  4. ArrayList:读快改慢，LinkedList:改快读慢。

二、总结

ArrayList和LinkedList是两个集合类，用于存储一系列的对象引用(references)。例如我们

可以用ArrayList来存储一系列的String或者Integer。那么ArrayList和LinkedList在性能上

有什么差别呢？什么时候应该用ArrayList什么时候又该用LinkedList呢？

  1. 对ArrayList和LinkedList而言，在列表末尾增加一个元素所花的开销都是固定的。对ArrayLi

    st而言，主要是在内部数组中增加一项，指向所添加的元素，偶尔可能会导致对数组重新进行分

    配；而对LinkedList而言，这个开销是统一的，分配一个内部Entry对象。

  1. 在ArrayList的中间插入或删除一个元素意味着这个列表中剩余的元素都会被移动；而在Linked

    List的中间插入或删除一个元素的开销是固定的。

  1. LinkedList不支持高效的随机元素访问。

  1. ArrayList的空间浪费主要体现在在list列表的结尾预留一定的容量空间，而LinkedList的空

    间花费则体现在它的每一个元素都需要消耗相当的空间

可以这样说：当操作是在一列数据的后面添加数据而不是在前面或中间,并且需要随机地访问其中的

元素时,使用ArrayList会提供比较好的性能；当你的操作是在一列数据的前面或中间添加或删除数

据,并且按照顺序访问其中的元素时,就应该使用LinkedList了。
