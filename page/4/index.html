Java中的equals和hashCode方法介绍

发表于 2018-03-08 | 更新于 2019-10-01 | 分类于 Java

一、equals()方法和hashCode()方法

Java中的equals()方法和hashCode()方法都是类Object中的方法。其中equals()方法用于检测一

个对象是否等于另外一个对象，默认是比较两个对象是否具有相同的引用。hashCode()方法是返回

该对象的散列码，默认是对象的存储地址。

二、equals方法具有的特性：

  1. 自反性:对于任何非空引用x,x.equals(x)应该返回true

  2. 对称性:对于任何引用x,y,当且仅当x.equals(y) 返回true，那么y.equals(x)也应该返回true

  3. 传递性：对于任何引用x,y,z;如果x.equals(y)返回true,y.equals(z)返回true,那么x.equals(z)也应该返回true

  4. 一致性:如果x和y引用的对象没有变化，那么x.equals(y)返回值永远不变

  5. 对于任何非空引用x,x.equals(null)都应返回false

三、hashCode方法

使用hashCode()方法查找一个集合中是否包含有某个对象时，不需要你的程序对集合中大量的元素

进行逐一比较，例如：当从HashSet集合中查找某个对象时，Java系统首先调用对象的hashCode()

方法获得该对象的哈希码表，然后根据哈希码找到相应的存储区域，最后取得该存储区域内的每个

元素与该对象进行equals()方法比较，从而提高了查找效率。

四、equals()方法和hashCode()方法之间的联系

1. 如果两个对象是相等的，那么他们的hashCode必须是相同的。

2. 如果两个对象具有相同的hashCode，它们可以相等，也可以不相等。

举例：

hashCode就好比字典里每个字的索引，equals()好比是字典里同一个字下的不同词语。就好像在字

典里查“自”这个字下的两个词语“自己”、“自发”，如果用equals()判断查询的词语相等那么就是同

一个词语，比如equals()比较的两个词语都是“自己”，那么此时hashCode()方法得到的值也肯定

相等；如果用equals()方法比较的是“自己”和“自发”这两个词语，那么得到结果是不想等，但是这

两个词都属于“自”这个字下的词语所以在查索引时相同，即：hashCode()相同。如果用equals()比

较的是“自己”和“他们”这两个词语的话那么得到的结果也是不同的，此时hashCode() 得到也是不

同的。

注意：如果重写了某类的equals()方法，那么肯定也需要重写该类的hashCode()方法，而且两

个对象的equals()返回值相等的情况下，这两个对象的hashCode()方法的返回值也必须相等。

Java容器

发表于 2018-03-07 | 更新于 2019-10-02 | 分类于 Java

一、概览

容器主要包括 Collection 和 Map 两种，Collection 存储着对象的集合，而 Map

存储着键值对（两个对象）的映射表。

1、Set:无序且不允许重复

常用方法：
1.add(E e) 如果 set 中尚未存在指定的元素，则添加此元素。

2.void clear() 移除此 set 中的所有元素。

3.boolean contains(Object o) 如果 set 包含指定的元素，则返回 true。

4.boolean equals(Object o) 比较指定对象与此 set 的相等性。

5.boolean isEmpty() 如果 set 不包含元素，则返回 true。

6.Iterator iterator() 返回在此 set 中的元素上进行迭代的迭代器。

7.boolean remove(Object o) 如果 set 中存在指定的元素，则将其移除。

8.boolean retainAll(Collection<?> c) 仅保留 set 中那些包含在指定 collection 中的元素(即求两个集合的交集)。

9.int size() 返回 set 中的元素数（其容量）。

10.Object[] toArray() 返回一个包含 set 中所有元素的数组。

2、list:有序且允许重复

常用方法：
1.boolean add(E e) 向列表的尾部添加指定的元素。

2.void add(int index, E element) 在列表的指定位置插入指定元素。

3.void clear() 从列表中移除所有元素。

4.boolean contains(Object o) 如果列表包含指定的元素，则返回 true。

5.boolean containsAll(Collection<?> c) 如果列表包含指定 collection 的所有元素，则返回 true。

6.boolean equals(Object o) 比较指定的对象与列表是否相等。

7.E get(int index) 返回列表中指定位置的元素。

8.int indexOf(Object o)返回此列表中第一次出现的指定元素的索引；
如果此列表不包含该元素，则返回 -1。

9.boolean isEmpty() 如果列表不包含元素，则返回 true。

10.Iterator iterator() 返回按适当顺序在列表的元素上进行迭代的迭代器。

11.int lastIndexOf(Object o) 返回此列表中最后出现的指定元素的索引；
如果列表不包含此元素，则返回 -1。

12.E remove(int index) 移除列表中指定位置的元素。

13.boolean retainAll(Collection<?> c) 仅在列表中保留指定 collection 中所包含的元素（求两个列表的交集）。

14.E set(int index, E element) 用指定元素替换列表中指定位置的元素。

15.int size() 返回列表中的元素数。

16.Object[] toArray() 返回按适当顺序包含列表中的所有元素的数组
（从第一个元素到最后一个元素）。

二、迭代器(Iterator)

Collection 继承了 Iterable 接口，其中的 iterator() 方法能够产生一个 Iterator

对象，通过这个对象就可以迭代遍历 Collection 中的元素，方法有：

  1. hasNext() :如果仍有元素可以迭代，则返回 true。
  2. E next() :返回迭代的下一个元素。
  3. void remove() 从迭代器指向的 collection中移除迭代器返回的最后一个元素。

三、collections类：用于操作集合类，如list，set。提供的所有方法都是静态的。

常用方法：

  1. sort方法的使用（对集合进行排序,默认按照升序排序，列表中所有元素必须实现Comparable
    接口）
  2. reverse方法的使用（对集合中元素进行反转）
  3. shuffle方法的使用（对集合中元素进行随机排序）
  4. binarySearch(Collection,Object)（查找指定集合中的元素，返回所查找元素位置的索引）

四、Comparable接口：实现此接口的对象列表（和数组）可以通过 Collections.sort（和 Arrays.sort）进行自动排序。

Comparable接口中只有一个方法：public int compareTo(T o);

调用此方法的对象，也就是this和o进行比较，若返回值大于0则this大于o，返回值等于0则是this

等于o，返回值小于0则是this小于o,而这个Comparable是直接在我们的自定义类上实现，因为thi

s是需要一个明确的比较对象的，

五、Comparator接口

1、Comparator接口中方法很多，但是我们只需要实现一个，也是最重要的一个int compare(T

o1,T o2);，也许有的人会好奇为什么接口中的方法可以不用实现，因为这是JDK8以后的新特性，

在接口中用default修饰的方法可以有方法体，在实现接口的时候可以不用重写，可以类比抽象类。

2、compare比较的o1和o2，返回值大于0则o1大于o2，依次类推，对于compare来说this是谁不重

要，所比较的两个对象都已经传入到方法中，所有Comparator就是有个外部比较器，在我们设计自

定义类初时，并不需要它有比较功能，在后期扩展业务是，Comparator的存在可以使我们在不修改

源代码的情况下来完成需求，只需要新定义一个比较器来实现Comparator，重写compare方法并将

自定义类对象传进去。

Comparable接口和Comparator接口总结

java中大部分我们常用的数据类型的类都实现了Comparable接口，而仅仅只有一个抽象类RuleBas

Java异常总结

发表于 2018-03-07 | 更新于 2019-09-15 | 分类于 Java

一、总结

二、概述

1、Java异常类图

  1. Error:JVM 系统内部错误、资源耗尽等严重情况

  2. Exception:

    1. RuntimeException（运行时异常）是指因设计或实现方式不当导致的问题。产生这种异常的特点是Java编译器不会检查它，也就是说程序，程序中可能出现这类异常时，即使没有用try-catch语句捕获它，也没有用throws语句声明抛出它，还是会编译通过的。由于没有处理它，当出现这类异常时，异常对象一直被传递到main()方法，则程序将异常终止。如果采用了异常处理，异常将会被相应的程序执行处理。
    2. CheckedException(检查异常)是RuntimeException以外的异常，类型上都属于Exception类及其子类。从程序语法角度讲是必须进行处理的异常，如果不处理，程序就不能编译通过。

  3. 用户自定义异常：如果要自定义异常类，则扩展Exception类即可，因此这样的自定义异常都属
    于检查异常（CheckedException）。如果要自定义非检查异常，则扩展自RuntimeException。
    按照国际惯例，自定义的异常应该总是包含如下的构造函数：

    • 一个无参构造函数
    • 一个带有String参数的构造函数，并传递给父类的构造函数。
    • 一个带有String参数和Throwable参数，并都传递给父类构造函数
    • 一个带有Throwable参数的构造函数，并传递给父类的构造函数。

2、五个关键字

  1. throws:用来标明一个成员函数可能抛出的各种”异常”。
  2. throw:用来明确地抛出一个”异常”。
  3. try:包含可能导致异常的保护代码。
  4. catch:捕获保护代码的异常。
  5. finally:确保一段代码不管发生什么”异常”都被执行一段代码。

throws和throw的区别
throws：用来声明一个方法可能产生的所有异常，不做任何处理而是将异常往上传，谁调用我我就抛给谁。

  • 用在方法声明后面，跟的是异常类名
  • 可以跟多个异常类名，用逗号隔开
  • 表示抛出异常，由该方法的调用者来处理
  • throws表示出现异常的一种可能性，并不一定会发生这些异常

throw：则是用来抛出一个具体的异常类型。

  • 用在方法体内，跟的是异常对象名
  • 只能抛出一个异常对象名
  • 表示抛出异常，由方法体内的语句处理
  • throw则是抛出了异常，执行throw则一定抛出了某种异常

3、先逮小的，再逮大的

即当捕获异常时，父类异常的捕获语句不可以写在子类异常的捕获语句的上面。

4、异常和重写的关系

  1. 父类没有声明异常，那么子类也不能声明异常；
  2. 重写的方法不能比被重写的方法抛出更宽泛的异常；
  3. 抛出异常类型的数目不可以比原有的方法抛出还多；

抽象类和接口

发表于 2018-03-07 | 更新于 2019-10-01 | 分类于 Java

一、抽象类（abstract修饰符）

  1. abstract修饰的类为抽象类，此类不能有对象(无法对此类进行实例化，即不能new)；
  2. abstract修饰的方法为抽象方法，此方法不能有方法体(就是什么内容都不能有)；
  3. 有抽象方法的类一定是抽象类，但抽象类中不一定有抽象方法；
  4. 抽象类中的抽象方法必须在子类中被实现，子类实现方法必须含有相同或更低的访问级别，
    抽象类的子类具体实现父类的所有抽象方法，否则也是抽象类；

二、接口(interface)

  1. 接口中的所有属性默认为:public static final;
  2. 接口中的所有方法默认为：public abstract;
  3. 接口不能被实例化；
  4. 接口只能包含方法声明；
  5. 接口不再像类一样用关键字extend去”继承”，而是用implements去”实现”;
  6. 一个类可以实现多个接口，但只能继承于一个类，所以可以通过这种方式来实现多继承；

三、接口和抽象类的区别

  1. 抽象类描述的是”是不是”的问题，而接口描述的是”有没有”的问题；
  2. 接口强调特定功能的实现，而抽象类强调所属关系；
  3. 抽象类中可以有构造方法，接口中不能有构造方法；
  4. 接口可以被多重实现，抽象类只能被单一继承；

四、接口和抽象类的相同点

  1. 都可以被继承
  2. 都不能被实例化
  3. 都可以包含方法声明
  4. 派生类必须实现未实现方法

一、面向对象语言三大特征

  1. 封装
  2. 继承
  3. 多态

二、设计思想

  1. 看到一个需求时不应该直接写代码，应该先考虑有哪些类；
  2. 考虑类的时候，类一定是一类事物的描述，不能太局限；
  3. 考虑类的时候需要考虑主要的类，也就是需要和业务、动作、事件紧密相关的类；
  4. 属性：不需要把该类的所有属性都声明出来，只需要自己用到的即可；
  5. 属性一般会私有，提供get、set方法区操作；
  6. 方法：考虑封装方法的时候，就站在该类的立场去考虑；
  7. 隐藏和封装：减低耦合度；
  8. 要增加或强化功能的时候，首先应该想的是扩展（重写或重载等），而不是修改；
  9. 当同等类型的、相似类型的类过多的时候，应该想到继承；
  10. 继承多了的时候可以采用多态的方式，用父类引用指向子类对象，从而解决了处理多
    个对象而产生的不同方法了
  11. 因为继承是很耦合性的，且继承只能继承一个类，所以如果要继承多个父类的情况就
    要用接口实现。

多态总结以及动态绑定机制

发表于 2018-03-06 | 更新于 2019-09-10 | 分类于 Java

一、多态

1、概念

父类引用指向子类对象（内存堆区的地址）

2、多态实现的必要条件

  • 存在继承关系
  • 存在方法重写
  • 父类引用指向子类对象

3、多态的体现

  • 继承体现了多态
  • 接口体现了多态

4、生活中的多态

关于多态的例子不胜枚举，比方说按下F1键这个动作，如果当前在Flash界面下弹出的就是

AS3的帮助文档；如果当前在Word下弹出的就是Word帮助；在Windows下弹出的就是Windows

帮助和支持。同一个事件发生在不同的对象上会产生不同的结果。

5、多态访问成员

  1. 访问成员变量

编译时期看左边的类型,如果左边类型中没有该变量则编译报错

程序运行时期看左边类型,左边类型的变量的值就是运行的结果

  1. 访问成员方法

编译时期看左边的类型,如果左边类型中没有该方法则编译报错

程序运行时期看右边类型,即程序会执行右边类型的方法

  1. 访问构造方法

多态访问子类构造方法时会先访问父类构造方法

我们可以利用这一特性来帮助子类初始化父类继承过来的成员

  1. 访问静态方法

编译时期看左边的类型,如果左边类型中没有该方法则编译报错

程序运行时期看右边类型,即程序会执行右边类型的方法

6、多态的优点

  • 简化代码
  • 提高程序的扩展性和可维护性

7、多态的缺点

使用父类引用无法直接访问子类所特有的成员，需要向下转型，即父类引用转为子类对象（强制转

换），且向下转型的过程中如果没有转换为真实子类类型，会出现类型转换异常（ClassCastExcep

tion），故每次在向下转型之前需要做一个类型判断。

二、动态绑定机制

1、概念

动态绑定是指在执行期间判断所引用对象的实际类型，根据其实际类型调用其相应的方法

2、程序绑定的概念

绑定指的是一个方法的调用与方法所在的类（方法主体）关联起来，对Java来说绑定分为静态绑定

和动态绑定，或者叫做前期绑定（编译时绑定）和后期绑定（运行时绑定）

3、区分

  1. 静态绑定

程序在JVM运行过程中，会把类的类型信息、static属性和方法、final常量等元数据加载到方法区

这些在类加载时就已经知道，不需要对象的创建就能访问的，就是静态绑定的内容；

  1. 动态绑定

需要对象创建出来，使用时根据队中的实例对象的类型才进行取用的就是动态绑定的内容。
